# EFoodPass SpringBoot → NestJS 1:1 复刻说明

## 1. 文档定位

这是一份 **用于指导将现有 SpringBoot 后端按 1:1 方式复刻为 NestJS 后端** 的执行文档。

这份文档的用途不是“设计一个新系统”，而是：

- 以现有 SpringBoot 项目为唯一业务基准
- 用 NestJS 重新实现一套功能完全相同的后端
- 尽量复用原有数据库结构与业务规则
- 保证前端视角下接口行为尽可能一致
- 便于后续保存进度、分阶段推进、让其他大模型辅助继续开发

本项目当前是**学习型复刻项目**，不是生产替换项目。

---

## 2. 核心目标

### 2.1 总目标

将当前 SpringBoot 版本的 EFoodPass 后端，完整复刻为一套 NestJS 后端，要求：

1. 功能范围与原 SpringBoot 保持一致
2. 数据库表结构保持兼容
3. 核心业务规则保持一致
4. 权限码保持一致
5. 主要接口入参、出参、错误行为尽可能一致
6. 可独立运行、独立调试、独立联调

### 2.2 本次复刻的本质

本次工作不是“从 Java 改写成 TypeScript 语法”这么简单，而是：

- 把 SpringBoot 中的模块划分复刻为 NestJS 模块划分
- 把 Spring Security + JWT 的认证鉴权思路复刻为 NestJS Guard 机制
- 把 Service 层业务逻辑复刻为 NestJS Service
- 把实体、DTO、分页、异常处理、统一响应这些基础设施完整迁移
- 把订单、库存、日志这种事务型业务在 NestJS 中原样实现

换句话说，目标是 **业务行为一致**，不是只求“看起来差不多”。

---

## 3. 复刻原则

## 3.1 以 SpringBoot 为唯一真源头

在复刻过程中，所有业务判断都以现有 SpringBoot 代码为准。

当出现以下冲突时，优先级如下：

1. 现有 SpringBoot 代码真实行为
2. 当前数据库结构与初始化 SQL
3. 项目交接文档 / README 等说明文档
4. NestJS 个人偏好实现方式

也就是说：

**NestJS 版不是重新设计，而是忠实翻译。**

## 3.2 先求一致，再求优化

第一阶段不追求“更优雅”“更高级”“更 Nest 风格”。

第一阶段目标只有一个：

**让 NestJS 版本行为与 SpringBoot 版本一致。**

只有在 1:1 复刻完成且通过验收后，才考虑后续优化，例如：

- Redis 增强
- 更细的模块抽象
- 更优雅的权限守卫封装
- 更强的库存并发控制
- 更规范的测试体系

## 3.3 不随意改数据库

本次复刻默认复用现有数据库表结构。

因此在 NestJS 中必须遵守：

- 不随意改表名
- 不随意改字段名
- 不随意改字段类型
- 不随意改枚举值 / 状态值
- 不随意新增、删除字段
- 不让 ORM 自动修改数据库结构

建议 ORM 配置中关闭自动同步。

## 3.4 不改变稳定业务标识

以下内容在复刻过程中必须视为稳定标识：

- username
- roleCode
- permCode
- 订单状态值
- 分类启用/禁用状态值
- 商品上架/下架状态值

尤其是 `permCode`，必须保持和原后端及初始化 SQL 一致。

## 5. 技术选型建议

## 5.1 推荐技术栈

建议 NestJS 版采用如下技术栈：

- NestJS
- TypeScript
- TypeORM
- MySQL
- Passport
- passport-jwt
- bcrypt
- class-validator
- class-transformer
- Swagger

## 5.2 为什么建议 TypeORM

因为本项目是“映射既有数据库”而不是“从零设计新库”。

TypeORM 在以下场景更贴近本项目需求：

- 明确映射现有表结构
- 便于定义实体与表字段对应关系
- 可使用事务能力
- 与 NestJS 集成成熟
- 更容易按 SpringBoot 的模块结构复刻

### 重要约束

- `synchronize: false`
- 不允许自动改表
- 若需要后续迁移，单独维护 SQL 或 migration

## 7. SpringBoot → NestJS 概念映射

| SpringBoot 概念 | NestJS 对应概念 | 说明 |
|---|---|---|
| Controller | Controller | 基本一一对应 |
| Service | Service | 一一对应 |
| ServiceImpl | Service | NestJS 一般不单独拆 impl |
| Entity | Entity | 若使用 TypeORM 则直接映射 |
| DTO / VO / Query | DTO | 使用 class-validator 管理 |
| GlobalExceptionHandler | ExceptionFilter | 做统一异常处理 |
| BusinessException | 自定义异常类 | 保留业务错误语义 |
| Result | Interceptor / 工具类 | 统一包装返回 |
| PageQuery / PageResult | DTO / 通用类 | 统一分页模型 |
| Spring Security | Guard + Strategy | 认证与权限控制 |
| @PreAuthorize | 自定义装饰器 + Guard | 权限码鉴权 |
| ThreadLocal 当前用户 | request.user | 挂载在请求对象上 |
| @Transactional | transaction / QueryRunner | 处理订单、库存等事务 |

## 8. 数据库复刻约束

## 8.1 数据库策略

建议保留两种环境：

### 方案 A：共用同一套测试库

适合快速验证接口行为一致性。

优点：

- 对照方便
- 数据一致
- 验证直接

风险：

- 两套后端同时写同一批数据，容易互相影响

### 方案 B：复制一套同结构数据库

更推荐学习场景使用。

优点：

- SpringBoot 和 NestJS 可独立调试
- 不会因调试误伤原数据
- 便于回滚与重置

建议：

**学习阶段优先使用独立库，但表结构必须完全一致。**

## 8.2 ORM 映射要求

所有实体字段都应与现有表结构严格对齐，包括：

- 主键字段名
- 时间字段名
- 状态字段名
- 外键关联字段名
- 逻辑含义字段

如果 SpringBoot 当前数据库中某些表已经有固定命名方式，NestJS 版必须按该命名映射。

## 9. 统一基础能力复刻要求

## 9.1 统一响应结构

NestJS 版必须建立统一响应格式，尽量保持与 SpringBoot 版一致，例如：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

具体字段名、成功码、错误码以原项目为准。

### 要求

- 所有成功响应统一包装
- 分页响应统一包装
- 异常响应结构统一
- 不允许每个 Controller 各写各的返回格式

## 9.2 分页结构

需要建立：

- `PageQueryDto`
- `PageResultDto<T>`

要求：

- 查询参数命名和原项目保持一致
- 分页结果字段命名尽量与原项目一致
- 排序逻辑保持一致

## 9.3 错误码体系

NestJS 版应建立与 SpringBoot 版对应的业务错误码常量。

要求：

- 保留业务语义
- 相同场景尽量返回相同错误码 / 相同 message
- 不要随意用 500 代替所有业务错误

## 9.4 全局异常处理

必须有全局异常过滤器，负责统一处理：

- 参数校验错误
- 业务异常
- 未认证异常
- 无权限异常
- 系统异常

目标：

前端看到的错误风格尽量和 SpringBoot 一致。

## 10. 认证模块复刻要求

## 10.1 功能目标

认证模块需至少复刻以下能力：

- 登录
- 返回 token
- 解析 token
- 获取当前登录用户
- 校验用户状态
- tokenVersion 校验

## 10.2 必须保持一致的行为

### 登录

- 根据用户名查用户
- 校验密码
- 校验用户是否启用
- 生成 JWT
- 返回必要用户信息与 token

### 当前用户信息

- 从 token 中识别当前用户
- 查询用户基础信息
- 返回角色/权限信息（若原项目如此）

### tokenVersion

- token 中应带上用户当前 tokenVersion 或可等价校验的信息
- 当用户改密码或被强制失效时，旧 token 应不可继续使用

### 多 token 并存

当前设计允许同一用户多个有效 token 并存。

NestJS 版也应保持这一行为，不要自行改成“单点登录”。

## 10.3 NestJS 设计建议

建议包含：

- `JwtStrategy`
- `JwtAuthGuard`
- `CurrentUser` 装饰器
- `AuthService`
- `AuthController`

## 11. 权限系统复刻要求

## 11.1 权限模型

当前系统是 RBAC 模型，核心关系为：

- 用户 ←→ 角色
- 角色 ←→ 权限
- 权限以 `permCode` 作为稳定标识

## 11.2 必须遵守的规则

- `permCode` 必须和原项目一致
- 禁止私自更名权限码
- 控制器权限校验必须与 SQL 初始化权限保持一致
- admin 相关保护逻辑要保留
- ADMIN 角色保护逻辑要保留
- 核心权限保护逻辑要保留

## 11.3 NestJS 实现建议

建议实现：

- `@RequirePermissions(...codes)` 装饰器
- `PermissionsGuard`
- 登录后将权限码集合挂到 `request.user`
- Guard 中统一检查当前用户是否具有权限码

### 注意

这里不要偷懒写成简单角色判断。

本项目的核心是 **权限码判断**，不是只看角色名。

## 15. 订单模块复刻要求

这是本项目复刻中最关键、最复杂的模块。

## 15.1 必须复刻的接口能力

### 后台订单

- 列表
- 详情
- 创建
- 处理
- 取消
- 完成

### 当前用户订单

- 列表
- 详情
- 创建
- 取消

### 订单统计

- 订单概览
- 状态统计
- 热销商品统计
- 每日金额统计

## 15.2 订单状态必须一致

订单状态值必须与原项目保持一致，例如：

- 10 待处理
- 20 处理中
- 30 已完成
- 40 已取消

不得自行重定义。

## 15.3 必须保留的核心业务规则

创建订单时：

- 校验用户存在且启用
- 校验商品存在
- 校验商品上架状态
- 校验商品所属分类启用状态
- 校验库存是否足够
- 同一请求中若出现重复商品，需先聚合数量再校验库存
- 创建订单后扣减库存
- 记录库存变更日志

取消订单时：

- 校验订单当前状态是否允许取消
- 回补库存
- 记录库存回补日志

用户侧查看订单时：

- 必须校验订单归属权

## 15.4 事务要求

订单创建、取消等操作必须使用事务。

目标是保证以下操作具有原子性：

- 订单主表写入
- 订单明细写入
- 库存扣减 / 回补
- 库存日志写入

任何一步失败，都必须整体回滚。

### 建议实现方式

可使用：

- TypeORM transaction
- QueryRunner

不建议将订单、库存、日志写成彼此独立无事务的调用。

## 17. 接口层复刻要求

## 17.1 URL 与方法

建议尽可能保持与 SpringBoot 版一致：

- 路由路径尽量一致
- HTTP 方法尽量一致
- 分页查询参数尽量一致

这样可以方便前端直接替换测试。

## 17.2 DTO 设计

NestJS 中应沿用 Java 项目中的思路：

- `Query` → 查询 DTO
- `CreateRequest` → 创建 DTO
- `UpdateRequest` → 修改 DTO
- `Response` → 返回 DTO

要求：

- 不直接把 Entity 裸返回给前端
- 参数校验尽量前置
- 与原项目字段语义一致

## 17.3 Controller 设计原则

Controller 只负责：

- 接收请求
- 参数绑定
- 权限声明
- 调用 Service
- 返回统一响应

业务逻辑必须放到 Service 中。

## 18. 日志、校验、异常建议

## 18.1 参数校验

所有 DTO 应尽量使用：

- `class-validator`
- `class-transformer`

确保：

- 必填项有明确校验
- 数值范围有校验
- 枚举 / 状态值合法性有校验

## 18.2 业务异常

不要把所有错误都抛成普通 `BadRequestException`。

建议建立专门的业务异常类，例如：

- 用户不存在
- 用户被禁用
- 分类不存在
- 商品不存在
- 库存不足
- 无权限
- 订单状态非法

## 18.3 系统日志

至少保留必要日志：

- 登录失败原因
- 关键业务失败位置
- 订单事务异常
- 权限鉴权失败原因

## 19. 推荐实施顺序

## 第一阶段：搭基础骨架

目标：把 NestJS 项目最基础的公共能力搭起来。

任务：

1. 创建 NestJS 项目
2. 配置 TypeScript / ESLint / Prettier
3. 接入 MySQL
4. 接入 TypeORM
5. 建立基础目录结构
6. 建立统一响应结构
7. 建立分页模型
8. 建立全局异常处理
9. 接入 Swagger
10. 建立基础错误码常量

验收标准：

- 项目可启动
- Swagger 可打开

## 22. 给辅助开发模型的使用说明

当把这份文档提供给其他大模型时，应明确要求对方遵守以下指令：

### 22.1 角色定位

你现在的任务不是重新设计一个新后端，而是：

**根据现有 SpringBoot 项目与这份复刻说明，按 1:1 原则复刻 NestJS 后端。**

### 22.2 输出要求

- 优先输出可执行代码
- 代码目录要清晰
- DTO、Entity、Service、Controller 要分层明确
- 不能跳过业务规则
- 不能随意篡改数据库字段和接口行为
- 若不确定，应优先参考现有 SpringBoot 逻辑

### 22.3 行为约束

- 不得擅自优化成另一套业务设计
- 不得擅自新增与原项目无关的复杂模式
- 不得删除原系统已有的业务校验
- 必须保持权限码、状态值、数据库字段含义一致
- 涉及订单、库存、日志联动时必须使用事务思维实现

### 22.4 最佳协作方式

建议把任务拆成小块让模型逐步完成：

1. 先生成目录结构
2. 再生成基础配置
3. 再做 auth
4. 再做 RBAC
5. 再做分类和餐品
6. 最后做订单和统计

不要一口气要求模型生成整个项目，否则很容易不稳定。

## 24. 最终结论

这次 NestJS 项目不是一个全新项目，而是对现有 SpringBoot 后端的**学习型、对照型、1:1 复刻项目**。

判断是否成功，不是看代码是不是更优雅，而是看以下几点：

1. 功能是否完整对齐
2. 数据库是否兼容
3. 权限是否一致
4. 核心业务规则是否一致
5. 订单库存事务是否一致
6. 前端联调时是否可替代 SpringBoot 版本

只要做到这些，这次复刻就是成功的。

后续若需要优化、重构、增强，那是第二阶段的事情。
